# 第九章：人工智能解决方案的安全、合规性和治理

摩根大通每年在网络安全上的支出超过 6 亿美元。1 美国银行的支出超过 10 亿美元。

这些并非例外。根据 Gartner 的研究，网络安全支出是一个庞大的类别，预计到 2024 年全球将达到 1830 亿美元。2 该公司预测，从 2023 年到 2028 年，复合年增长率（CAGR）将达到 11.7%。推动这一增长的因素包括人工智能和云计算技术的威胁增加。

对于 AWS 来说，安全是其首要任务。这包括对保护平台的大量投资，以及为数百万客户提供广泛的服务。其中许多服务——如 AWS IAM、Amazon GuardDuty 和 AWS Config——在 AIF-C01 考试中被重点介绍。在本章中，我们将探讨这些工具以及人工智能系统的治理和安全更广泛的原则——这些是通过 AWS 考试的关键主题。

# 安全、合规性和治理概述

通常会将安全、治理和合规性这些概念混为一谈。然而，每个概念都有其不同的角色，理解这些区别非常重要。以下是详细说明：

安全

重点关注保护数据和基础设施，其目标是确保机密性（只有正确的人才能访问信息）、完整性（数据保持准确和可信）以及可用性（在需要时系统和数据可访问）。你通常会听到这个功能被称为*信息安全*或*网络安全*。

治理

这关乎于明智地引导组织。它确保业务在有效管理风险的同时能够创造价值。在人工智能的背景下，治理提供了保持创新与问责制一致的结构。它帮助组织快速行动而不会破坏事物——尤其是当这些“事物”包括客户信任、道德标准或合规性时。

合规性

这确保了组织遵循规则——无论这些规则来自法律、法规、内部政策还是行业标准。这是关于持续和可靠地满足要求。

这三个功能共同帮助组织实现其核心使命。它们定义了不可协商的事项——那些不应被妥协的基本保障。

对于本章的其余部分，我们将为每个功能设置一个部分，并介绍考试中你需要了解的每个功能的内容。

## 安全

为了有效地保护人工智能系统，组织采用分层策略和特定框架来管理不同用例下的风险。

### 使用 AWS 工具的安全方法

*深度防御* 是一种基于单一控制无法万无一失这一理念的多层安全方法。想想看，就像保护你的家一样。你可能锁上了前门——这就是你的防火墙，比如 AWS 网络应用防火墙（WAF）或安全组。但你也安装了报警系统，这类似于使用 Amazon GuardDuty 或 AWS Security Hub 来检测和警告威胁。添加运动探测器，你就可以主动感知异常活动——就像使用 Amazon EventBridge 触发自动化响应一样。最后，你设置了安全摄像头，记录事件以便你可以回顾发生了什么。在 AWS 中，这是 CloudTrail 或 AWS Config，它们提供日志和历史可见性。如果一项措施失败，其他措施可以帮助捕捉漏网之鱼。这是 AWS 上深度防御的本质。

这种策略——很可能会在考试中出现——在处理生成式 AI 时变得尤为重要，因为工作负载通常涉及敏感数据、有价值的知识产权和快速发展的开发周期（见图 9-1）。

在设置层之前，你应该编写清晰、可执行的政策。例如，假设你的数据科学团队经常启动训练集群。使用 AWS IAM 实施最小权限访问，并使用访问分析器标记可能过于宽泛的权限。然后强制执行短期凭证，以确保没有人获得他们不需要的长期管理访问权限。

让我们更详细地看看这些层：

数据保护

对于静态数据，你可以使用 AWS 密钥管理服务（KMS）来加密一切——从 Amazon S3 中的训练数据集到模型检查点。在 Amazon S3 中启用版本控制，以便在数据被损坏或篡改时可以回滚。

对于传输中的数据，使用 AWS 证书管理器（ACM）处理 TLS 证书，并使用 AWS Private CA 签发内部证书。通过 AWS PrivateLink 路由敏感流量，以避免将其暴露在公共互联网上。

IAM

使用 IAM，你应该为你的模型训练、推理和监控工作负载创建不同的角色。避免使用根凭证，并为每个人类用户启用多因素认证（MFA）。

应用程序保护

有不同的方法来保护应用程序。例如，你可以使用 AWS Shield 来减轻拒绝服务（DoS）攻击，并使用 Amazon Cognito 来安全地管理用户登录和身份联合。

假设你正在托管一个生成文本 API。一个恶意行为者可能会尝试通过自动化请求来压倒它。Shield 帮助吸收流量，而 Cognito 通过实施速率限制和身份验证规则来保持访问安全。

威胁检测和事件响应

事情总会出错。关键是尽早发现问题并知道如何应对。你可以使用 Amazon GuardDuty 在你的账户中检测可疑活动。结合 AWS Security Hub 来集中跨服务的警报。

当发生事件时，自动化你的第一步。使用 Amazon EventBridge 触发一个 Lambda 函数，隔离可疑的 Amazon EC2 实例或自动撤销 IAM 权限。这为你的团队提供了更多时间来调查。

基础设施保护

作为深度防御策略的一部分，基础设施层充当了关键防线之一。在这里，目标是通过对访问进行控制和隔离资源来加固你的环境以抵御潜在的攻击。例如，你可以使用 IAM 策略来严格管理谁被允许启动或修改基础设施组件。网络访问控制列表（ACLs）通过限制子网之间的流量流来增加另一层防护，帮助在某个区域被入侵时遏制威胁。此外，定义具有明确边界的 IAM 用户组确保只有授权的角色可以执行敏感操作。

网络和边缘保护

你希望在网络的边缘建立强大的保护。你可以使用 Amazon VPC 为你的 AI 管道每个阶段创建隔离的环境。添加 AWS WAF 在边缘阻止常见的漏洞利用，并尽可能限制对公共端点的访问。

### 生成式 AI 安全范围矩阵

生成式 AI 安全范围矩阵是 AWS 开发的一个框架，旨在帮助组织评估和实施其生成式 AI 工作负载的安全控制。它将 AI 实现分为五个不同的范围，每个范围代表对 AI 模型和相关数据的控制责任的不同水平（参见图 9-2）。

让我们更详细地看看每个范围：

范围 1：消费应用

在这个层面，你直接使用公开可用的生成式 AI 工具——无需定制，无需后端访问。这些工具如 ChatGPT 或其他类似平台，任何人都可以通过注册或登录来使用。你完全在提供商的环境中操作，无法了解模型是如何训练的或它使用了哪些数据。所有交互都通过用户界面或 API 进行，并且你受限于提供商的服务条款。一个典型的例子是员工向 ChatGPT 请求营销活动的创意想法。

范围 2：企业应用

在这里，你正在使用为商业量身定制的第三方软件，其中包括嵌入式生成式 AI 功能。与范围 1 不同，这些工具通常带有正式的供应商关系——合同、支持和服务级别协议（SLA）。你可能获得更多的灵活性或配置选项，但核心 AI 模型仍然牢牢掌握在供应商手中。一个很好的例子是使用一个商业级日历应用，该应用利用 AI 根据你的过去模式和输入来起草会议议程。

范围 3：预训练模型

在范围 3 中，你正在使用预存在的 AI 模型构建自己的应用程序，这些模型通过 API 访问。模型本身存在于第三方平台上，但你控制着它如何融入你的应用程序、你向它输入什么数据以及它如何与你的业务流程互动。例如，你可能会创建一个客户服务聊天机器人，该机器人通过 Amazon Bedrock 连接到 Anthropic 的 Claude 模型，根据你的输入数据和应用程序流程定制响应。

范围 4：微调模型

在这个范围内，你正在使用自己的数据微调 FM，以更好地满足你的特定需求。这赋予你更多的控制权，同时也增加了更多的责任——你将根据你专有的信息塑造模型的行为。假设你的营销团队希望生成反映你品牌声音的内容。你可以使用过去的营销活动、客户数据和语气指南来微调现有的模型，以生成精准的促销材料。

范围 5：自训练模型

在最高控制级别，你从头开始构建和训练自己的生成式 AI 模型。这意味着收集数据、设计架构、运行训练并维护整个系统。你拥有一切——数据、模型、结果——并对性能、伦理和合规承担全部责任。这个范围适合具有高度专业化需求或严格监管环境的组织。例如，一家金融公司可能会开发自己的 FM，该 FM 专门在监管文件、内部报告和市场数据上进行训练，以支持授权的分析服务。

对于每个范围，AWS 强调五个关键的安全学科：

治理和合规

建立政策和程序来管理风险并确保遵守法规

法律和隐私

确保符合法律要求并保护用户数据隐私

风险管理

识别潜在威胁并实施缓解策略

控制

应用适合责任和风险级别的安全控制

弹性

设计系统以保持可用性并从中断中恢复

### AI 和生成式 AI 的安全

当涉及到 AI 和生成式 AI 时，从保护基础设施到防范输入操纵，攻击面以新的和有时意想不到的方式增长。让我们分析五个对于减轻这些威胁至关重要的基本领域：

威胁检测

在 AI 背景下，威胁检测意味着积极监控可能有人或某些东西试图破坏你的系统的迹象。攻击者可能会使用生成式 AI 来创建虚假内容、篡改数据或自动化更广泛的网络攻击的一部分。为了跟上，你可以部署 AI 驱动的工具，这些工具可以筛选网络流量、分析用户行为并监视异常模式。

漏洞管理

每个系统都有弱点，AI 也不例外。代码中的错误、模型中的缺陷以及可利用的入口点——如恶意软件感染的文件或钓鱼附件——都可能造成风险。管理这些漏洞意味着进行定期的安全评估、进行渗透测试（故意尝试破坏自己的系统）以及进行详细的代码审查。同样重要的是，要关注补丁和更新。未打补丁的软件是攻击者入侵系统最简单的方法之一。

基础设施保护

人工智能依赖于云平台、边缘设备、数据库和其他基础组件。如果这些部分不安全，那么你的 AI 系统也同样不安全。基础设施保护包括设置严格的访问控制、通过网络分段隔离系统以及加密敏感资源。你还希望设计具有弹性的基础设施——这样它就能快速从攻击、中断或系统故障中恢复，而不会使你的 AI 系统离线。

提示注入抵抗

提示注入是一种较新的威胁，是生成式 AI 独有的。为了防御这种威胁，对所有传入的提示进行清理和验证。你还可以设计更抵抗这种操纵的模型和训练过程——本质上就是教会系统忽略可疑的指令。

数据加密

使用强大的加密来保护数据，无论是存储时（静止数据）还是系统间传输时（传输数据）。同样重要的是，要小心管理你的加密密钥。如果攻击者能够访问它们，加密就变得毫无用处。把它们当作你整个运营的钥匙来对待——因为在许多方面，它们就是。

### AWS 为 AI 工作负载提供的安全工具和服务

AWS 提供了许多安全工具，帮助你从数据、基础设施到身份和访问的每一层保护 AI 系统。你已经看到了像 AWS 安全中心（AWS Security Hub）和 Amazon GuardDuty 这样的服务如何集中和自动化威胁检测。让我们看看一些其他的服务，这些服务可以完善你的 AI 安全工具箱。

#### AWS 密钥管理服务

你可以选择使用 AWS 管理的密钥以方便使用，或者如果你需要更紧密的控制，可以创建和管理自己的客户密钥。无论哪种方式，KMS 都能确保你的数据保持安全，无论是处于静止状态还是在传输过程中。

#### AWS Shield Advanced

它包括 AWS WAF 和 AWS Firewall Manager 等工具，为你提供分层防御和集中式策略管理。

#### Amazon Macie

亚马逊 Macie 利用机器学习自动识别和分类您 AWS 环境中的敏感数据。它特别有助于扫描 S3 存储桶以发现 PII（个人身份信息）、受保护的医疗信息（PHI）或财务记录。如果您正在为 AI 模型准备训练数据集，Macie 可以标记需要删除或进一步保护的数据。您甚至可以通过将数据库内容导出到 S3 并扫描这些数据来扩展此功能。

#### 零信任和细粒度访问控制

要采用零信任方法，AWS 提供了如验证访问和验证权限等工具。这些服务允许您实施细粒度的访问策略，而不依赖于传统的 VPN。它们帮助您以可扩展和高效的方式实施基于身份的安全性。

#### 亚马逊 SageMaker 角色管理器

如果您使用亚马逊 SageMaker，角色管理器可以帮助您创建针对不同机器学习角色的 IAM 角色。它包括内置的角色——如数据科学家、MLOps 工程师和 SageMaker 计算——这些角色为常见任务预配置了权限。

#### 网络安全和数据流控制

您可以使用 AWS 网络防火墙和 Amazon VPC 策略在网络级别管理数据流入和流出。AWS 网络防火墙支持深度数据包检查，让您在数据离开或进入您的环境之前解密并检查 TLS 流量。Amazon VPC 让您完全控制您的虚拟网络环境，类似于管理自己的本地数据中心。为了避免将内部流量暴露给互联网，AWS PrivateLink 允许您将 VPC 私密地连接到如亚马逊 Bedrock 等服务（见表 9-1）。

## 合规性

人工智能的合规性很复杂，因为它必须与不断发展的法律、伦理和技术期望保持一致。

第一步是建立一个 AI 治理委员会或小组。这个团队应该包括来自组织各个部门的人——不仅仅是技术和数据科学，还包括法律、风险、合规、隐私，甚至客户倡导。你需要的是一群既理解技术又理解其潜在影响的人。

假设你正在构建一个用于筛选求职者的 AI 模型。你的 AI 团队可能专注于性能指标，但人力资源和法律部门可以帮助早期发现偏见或公平性问题。从第一天起让他们参与其中可以避免以后的麻烦。

一旦委员会成立，就要明确界定角色和责任。在部署前谁负责审查模型？如果生产中出现问题，谁负责处理升级路径？谁负责制定关于可以使用和不能使用的数据类型的政策？

然后转向政策和程序，例如以下主题：

+   数据来源和隐私要求

+   模型训练和评估标准

+   部署标准和审批步骤

+   对漂移、滥用或监管变化的持续监控

### 合规性标准

AWS 支持超过 140 个安全标准和合规性认证。诚然，每个客户都有权决定他们愿意承担多少风险。但有一些安全框架在处理 AI 系统时特别相关：

国家标准与技术研究院（NIST）

NIST 800-53 框架概述了一套主要用于美国联邦机构的安全控制措施。遵循此标准的组织将进行正式评估，以确认他们已经实施了适当的安全保护措施来保护敏感信息。

欧洲联盟网络安全局（ENISA）

ENISA 在塑造欧盟的网络安全方法中扮演着核心角色。它开发了建立对数字服务和基础设施信任的认证结构，并与欧盟成员国紧密合作，帮助准备应对不断发展的网络威胁。

国际标准化组织（ISO）

ISO 安全标准——特别是基于 ISO/IEC 27002 的标准——为管理安全风险提供路线图。它们提供了最佳实践和详细控制措施，用于创建强大的信息安全管理系统（ISMS）。

AWS 系统和组织控制（SOC）

AWS 的 SOC 报告是第三方审计，验证 AWS 如何实施其合规性和安全实践。

健康保险可携带性和问责法案（HIPAA）

对于美国的医疗保健组织，AWS 通过提供处理 PHI 的安全环境来支持 HIPAA 合规性。这包括从存储到处理和传输数据的各个方面。

一般数据保护条例（GDPR）

GDPR 为欧盟的数据隐私设定了高标准。它旨在保护个人信息，并赋予欧盟居民对其数据使用的更多控制权。

信用卡行业数据安全标准（PCI DSS）

PCI DSS 是关于保护信用卡数据的。由 Visa 和万事达卡等主要信用卡公司组成的 PCI 安全标准委员会管理，它概述了保持支付数据安全的技术和操作要求。

合规性很复杂，因为它需要与人工智能构建、部署和使用方面的法律、伦理和技术期望相一致。而且与传统的软件不同，人工智能带来了一些新的复杂因素。

以下是一些人工智能合规的挑战：

复杂性和缺乏透明度

人工智能系统——尤其是大型语言模型和生成式人工智能——通常像黑盒一样运行。它们的内部逻辑可能极其复杂，而且并不总是清楚它们是如何生成特定输出的。这种缺乏可解释性使得合规审计更加困难。

持续变化

许多模型随着时间的推移而演变，通过学习新数据或在生产中进行调整。这对传统的合规框架来说是个问题，因为它们通常期望系统一旦部署就会保持一致的行为。

演化能力

随着人工智能系统变得更加复杂，它们可以发展出演化能力——即设计者没有计划到的技能或行为。这些不是错误或有人明确编码的功能。它们是复杂系统交互的副产品。这种不可预测性意味着监管者和开发者都需要保持警觉和灵活。

责任

人工智能系统需要是可解释的、可追溯的，并接受人类审查。一些政府已经开始朝这个方向发展。例如，欧盟的人工智能法案规定了透明度、风险评估和人类监督的要求。在美国，纽约等城市已经通过了要求披露和审查自动化决策工具的法律。

### 规范化的工作负载

规范化的工作负载是指必须遵循特定合规规则的工作负载——无论是法律、行业特定还是与安全和责任问题相关。这些要求通常适用于医疗保健、金融或航空航天等领域，在这些领域，系统处理敏感数据或影响高风险决策。如果你的工作负载属于 HIPAA、GDPR、PCI DSS 或 FDA 法规等标准，那么它显然是受规范的。

然而，规范并不总是看起来一样。它可以体现在你的运营方式、你管理的风险或你的系统所需的监督程度。以下是一些例子：

监管下的流程

例如向 FDA 等机构提交报告

带有后果的决策

如抵押贷款审批或信用评分，其中公平性和透明度至关重要

关键系统使用

在可能危及生命、健康或基础设施的领域

人工智能模型的法律责任

尤其是当模型输出可能导致法律或财务后果时

一些合规期望并非法律强制，但仍需关注。例如，HIPAA 为组织如何治理数据设定了政策——而不仅仅是它们如何存储或传输数据。这些标准可能通过审计、合同义务或行业标准来执行。通常需要密切监管的工作负载包括：

+   处理机密员工数据的 HR 系统

+   影响人类健康或公共安全的监控系统

+   用于审计或内部控制的合规和检查工作流程

并非每个工作负载都带有法律标签，因此提出正确的问题很有帮助：

+   这个工作负载是否需要接受审计？

+   我是否需要保留数据一段时间？

+   输出是否被视为官方记录或特殊数据类型？

+   这个工作负载是否触及具有内部分类规则的数据——即使不是正式受监管的？

如果您对这些问题的任何一个回答是肯定的，请将您的 workload 视为受监管。谨慎合规总比后来措手不及要好。

### AWS 合规工具

AWS 中的合规工具旨在帮助您满足监管、行业和内部标准。这些服务简化了证据收集流程，提供第三方审计的访问权限，并识别可能构成合规风险的漏洞。

#### AWS 审计管理器

AWS 审计管理器通过持续评估您的 AWS 环境与预构建或自定义控制框架（如 SOC 2、GDPR、HIPAA 和 ISO 27001）的对比，自动化证据收集过程。它从 AWS 服务中收集和映射数据，因此您可以以更少的手动工作生成审计准备好的报告。这有助于减轻审计准备工作的负担，并确保您的控制措施按预期运行。

#### AWS Artifact

AWS Artifact 是一个自助门户，用于访问 AWS 的合规报告和认证。它包括诸如 SOC 1/2/3 报告、ISO 认证和 PCI 合规文档等文件。AWS Artifact 帮助您了解 AWS 如何符合各种标准，但您仍然负责配置和管理您的环境以满足您自己的合规义务。

#### Amazon Inspector

Amazon Inspector 是一种漏洞管理服务，它持续扫描您的 EC2 实例、Lambda 函数和容器镜像以查找已知的安全问题。它使用来自国家漏洞数据库（NVD）等来源的现实世界威胁情报来识别软件漏洞和网络暴露。发现结果根据严重性进行优先排序。这允许首先解决高风险问题——这是维护需要定期漏洞评估的标准的重要步骤。

#### AWS Trusted Advisor

AWS 可信顾问评估您的 AWS 账户是否符合安全、容错、性能、服务限制和成本优化的最佳实践。从合规性的角度来看，其安全检查——如暴露的端口或过于宽松的 IAM 策略——有助于组织主动识别和解决可能导致合规违规的风险（见表 9-2）。

### 数据治理概念

数据治理是更广泛治理伞下的一个专业领域。它专注于数据如何收集、存储、访问、保护和使用。它涉及政策、标准、角色和工具，以确保数据质量、完整性、安全性和隐私。

Amazon 强调了六个关键数据管理概念，这些概念在 AI 系统的开发、部署和持续健康中发挥着至关重要的作用（见图 9-3）。

让我们逐一看看它们：

数据生命周期

数据生命周期概念描述了数据从创建到归档或删除的整个过程。对于 AI 工作负载，这包括从原始数据收集到数据处理、存储、用于训练和推理，以及最终退役的各个方面。

每个阶段都需要仔细规划。例如，如果在初始阶段收集数据时没有清晰的标签，那么在训练模型时可能会遇到严重问题。另一方面，未能归档或妥善处理过时数据可能会引入合规风险或增加存储成本。

数据记录

数据记录是关于详细记录您的 AI 系统如何处理其数据。这包括捕获输入和输出数据、模型性能指标和关键系统事件。这有助于您调试问题、监控持续性能，并在发生意外情况时追踪问题。

数据记录是透明度和问责制的一个强大工具。当你试图理解为什么模型做出了某个预测——或者为什么它突然开始表现不佳时——日志通常可以讲述整个故事。

数据驻留

数据驻留指的是你的数据物理上驻留的位置以及它被处理的位置。一些国家要求关于其公民的数据留在其境内——这一概念被称为数据主权。

从人工智能的角度来看，数据驻留决策也会影响性能。将你的训练数据保留在执行繁重计算的资源附近可以减少延迟并降低成本。但这里的主要启示是，你需要知道你的数据在哪里，以及为什么它在那里，尤其是如果你在多个区域或云服务提供商之间工作。

数据监控

数据监控是持续监控你的数据质量、一致性和相关性的实践。随着时间的推移，现实世界的数据会发生变化，你的模型可能会变得不那么有效。

监控有助于在这些问题导致性能问题之前捕捉到异常、低质量数据或分布变化。对于管理生产人工智能系统的团队来说，这种监督对于防止模型无声失败并保持对系统输出的信任至关重要。

数据分析

数据分析涉及分析你的数据集以了解其结构，检测模式，并揭示可以塑造你如何训练和评估模型的见解。这通常包括统计摘要、数据可视化和探索性数据分析（EDA）等方法。

没有适当的分析，你可能会给你的模型提供不完整、有偏见或不相关的数据。良好的分析不仅可以提高模型设计和特征工程，还有助于识别可能导致模型预测盲点的数据集缺口。

数据保留

数据保留关乎决定保留你的数据多长时间——以及原因。在人工智能领域，保留策略可以服务于多个目的，例如满足法律或行业标准，保留历史数据以进行再训练，或管理云存储的成本。

这是一个平衡行为。保留数据时间过长可能会增加隐私和合规性方面的风险，而丢弃数据过快可能会消除随着时间推移改进模型所需的宝贵上下文。

### AWS 治理工具和服务

AWS 中的治理侧重于管理大规模资源，执行组织政策，并确保多个账户和团队之间配置的一致性。这些工具帮助组织保持控制，标准化环境，并使云使用与业务和监管要求保持一致。

#### AWS Organizations 和服务控制策略（SCPs）

AWS Organizations 允许集中管理多个 AWS 账户，使您能够从单一位置分组账户、应用策略和管理账单。服务控制策略（SCPs）作为权限防护措施，定义了在特定账户或组织单元内可以或不可以执行的操作——无论个人 IAM 权限如何。这有助于防止在您的环境中意外或未经授权使用敏感服务或配置。

#### AWS Control Tower

AWS Control Tower 提供了一种设置和治理安全、多账户 AWS 环境的方法，也称为*着陆区*。它自动化账户创建，配置防护措施，并设置日志和安全基线。使用 Control Tower，企业可以在定义的边界内快速启用开发团队，同时保持一致的政策。

#### AWS Config

AWS Config 跟踪并记录您 AWS 资源的配置更改。这使您能够随着时间的推移评估与期望状态的合规性。它提供了资源配置和关系的详细历史记录，有助于故障排除、审计准备和政策执行。您还可以定义 Config 规则，以自动检测不合规的资源并触发修复。

#### AWS CloudTrail

尽管 CloudTrail 通常被归类为安全审计工具，但它通过记录您 AWS 账户中的每个 API 调用，也发挥着治理作用。它使您能够了解用户和服务活动，支持合规性调查，并可以根据特定操作触发警报。CloudTrail 日志对于维护责任和跨分布式团队执行治理至关重要（见表 9-3）。

### 理解数据和模型血缘

数据和模型血缘指的是您的数据和模型从何而来、随时间如何变化以及塑造它们的流程的完整历史。在人工智能——特别是在生成式人工智能中——跟踪这种血缘至关重要。它为您提供了系统起源、可靠性和数据或模型中任何潜在偏差的清晰图景。

在 AWS 中理解数据和模型谱系涉及所有三个类别——合规性、安全性和治理——但它与治理直接相关，与合规性有着紧密的联系。

接下来，我们将探讨数据和模型谱系的关键组成部分。

#### 源引用和数据来源文档

源引用和数据来源文档对于构建可信赖的 AI 至关重要。它们是帮助使你的系统透明、可追溯和可问责的实践。以下是对每个部分更详细的了解：

源引用

在生成式 AI 中，源引用意味着正确地承认你的训练数据来自哪里。无论你是从数据集、数据库或其他资源中提取，清楚地记录每个来源都很重要。你还将想要捕捉与数据相关的任何许可证、权限或使用条款。

记录数据来源

记录数据来源更加深入。它涉及到记录关于训练数据如何收集、整理、清洗和转换的每一个细节。以下是你应该记录的内容：

+   数据是如何和在哪里收集的

+   数据是如何清洗和整理的

+   任何预处理步骤或转换

通过这样做，你可以早期揭示任何隐藏的偏见、局限性或质量问题。这种洞察力可能会决定你模型未来的可靠性。

#### 数据谱系

当做得正确时，数据谱系会使你的源引用和起源文档变得更加容易。我们将探讨一些主要的方法：

编目

编目系统地组织你的数据集、模型和资源。把它想象成为你的 AI 系统建立一个图书馆：每一份数据、每一个模型和每一个许可证都有一个包含所有详细信息的“书籍”。

一个维护良好的编目可以改善你管理、沟通和审计数据及模型谱系的方式——无论是内部还是与外部利益相关者。

模型卡片

模型卡片提供了一种标准化的方式来记录你的机器学习模型。在生成式 AI 中，一个好的模型卡片会讲述以下全部内容：

+   模型使用了哪些数据

+   那些数据来自哪里

+   与数据相关的任何许可证或条款

+   已知的偏见、风险或质量问题

除了数据来源之外，模型卡片还描述了模型的目标用途、性能基准和局限性。它们帮助你向用户设定正确的期望，支持审计，并将你的模型与业务目标对齐。

如果你使用 Amazon SageMaker，SageMaker 模型卡片可以通过提供一个集中空间来存储所有模型详细信息，使这一过程更加顺畅。

### 生成式 AI 中数据使用的回顾

有效的数据治理始于理解生成式 AI 中使用的不同类型的数据及其控制者。大多数生成式 AI 应用依赖于三个关键类别：用户数据、微调数据和训练数据。每个类别在塑造模型性能方面都发挥着独特的作用——并且每个类别都伴随着不同的治理影响。

让我们更详细地看看每种类型的数据及其通常的治理方式。

#### 用户数据

用户数据包括客户或最终用户提供的任何内容——输入、提示、要求——基本上，用户发送到系统中以生成特定输出的任何内容。

无论应用范围如何，客户始终控制自己的用户数据。这是一个重要的常数，您可以信赖。

#### 微调数据

微调数据用于调整预训练的生成式 AI 模型以满足客户或特定领域的特定需求。

这就是微调数据通常是如何工作的：

+   它通常是原始训练数据的一个子集或从应用程序领域收集的新数据。

+   微调调整模型的内部设置——其参数和权重——以便为手头的任务产生更相关、更个性化的结果。

微调数据的控制取决于应用范围：

+   在范围 1 和 2 中，应用程序提供商控制微调数据。

+   在范围 4 中，客户控制微调数据。

了解谁控制什么对于治理和合规性都至关重要，因此值得牢记这一点。

#### 训练数据

训练数据是用于构建模型初始知识和核心能力的大型、多样化的数据集。

这是您需要了解的训练数据：

+   它通常包括广泛的内容——文本、图像、音频——具体取决于模型设计要做什么。

+   这些数据教会模型生成新、有意义输出所需的模式、结构和关系。

当涉及到所有权时：

+   在范围 1、2、3 和 4 中，应用程序提供商控制训练数据。

+   在范围 5 中，客户控制训练数据。

控制在如何处理不同生成式 AI 项目中的治理、安全和定制方面发挥着重要作用。

### 安全数据工程

强大的安全数据工程实践对于安全、可靠的 AI 和生成式 AI 系统至关重要。让我们深入了解您需要关注的重点领域。

#### 评估数据质量

评估数据质量始于设定清晰的指标。您需要定义完整性标准，确保您的训练数据覆盖广泛的代表性场景，没有重大缺口或偏差。准确性同样重要；数据必须正确、及时，并反映模型将面临的真实世界情况。及时性，有时也称为*货币性*，衡量数据的新鲜程度——过时的数据会迅速削弱模型的表现。最后，一致性确保数据在整个开发和部署过程中保持一致性和逻辑性。为了执行这些标准，在数据管道的多个阶段集成验证检查，定期进行数据概要分析，并在出现质量问题时进行监控。维护一个反馈循环以持续改进，并记录详细的数据来源和元数据，以跟踪数据旅程和转换，这也是至关重要的。

#### 实施增强隐私的技术

保护用户和训练数据需要实施增强隐私的技术。从数据掩码、数据模糊化或差分隐私等技术开始，这些技术有助于降低即使发生泄露，暴露敏感信息的风险。通过使用加密、令牌化和安全多方计算来保护数据，在处理或存储过程中进一步强化你的防御。这些方法共同工作，确保数据得到保护，同时不牺牲性能或可用性。

#### 数据访问控制

控制谁可以访问你的数据以及在什么情况下访问是维护安全的基本要求。建立一个强大的数据治理框架，具有明确的政策来规范访问、使用和共享。实施基于角色的访问控制，并分配细粒度的权限，以确保用户只能访问他们真正需要的。通过使用单点登录（SSO）、多因素认证（MFA）和 IAM 解决方案等身份验证和授权系统来加强这些控制。通过监控和记录所有数据访问活动来密切关注你的系统，以早期捕捉未授权的使用。定期审查和更新权限，以符合最小权限原则，确保每个角色都有必要的最小访问权限。

#### 数据完整性

维护数据完整性确保你的 AI 模型建立在坚实、可信的基础上。在整个数据管道中实施验证和完整性检查，例如模式验证、引用完整性检查和业务规则验证，以在错误造成更大问题之前捕捉到它们。始终准备好强大的备份和恢复策略，以便在系统故障、错误或灾难发生后快速恢复数据。使用事务管理和原子性原则，在处理和转换过程中保持数据的一致性和可靠性。通过维护详细的数据血缘和审计跟踪来记录数据的完整历史，这允许你跟踪每一个变化。最后，养成定期监控和测试你的数据完整性控制的习惯，根据需要调整它们，以保持对不断发展的风险的弹性。

# 结论

保护 AI 系统是必要的。随着组织构建更复杂的模型并采用生成式 AI 技术，风险也在不断上升。安全、合规和治理必须携手合作，以保护敏感数据，确保道德实践，并满足不断增长的监管需求。通过采取分层防御方法，建立明确的治理框架，并实施强大的数据管理实践，公司不仅可以降低风险，还可以增强客户、合作伙伴和监管机构的信任。最终，构建负责任的 AI 不仅仅是避免问题，而是创造具有弹性、透明和面向未来的解决方案。
