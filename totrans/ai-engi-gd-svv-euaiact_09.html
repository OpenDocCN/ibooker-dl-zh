Appendix B. Data Products and Data Contracts

Data products and their contracts are becoming popular as lightweight mechanisms for scalable data management, helping ensure that data quality is not an afterthought but a fundamental consideration at every stage of the lifecycle.

Data Products

Data products are curated outputs derived from raw data that provide value to end users, typically in the form of insights, actionable information, or analytics. They are created through various stages of the data engineering lifecycle, including ingestion, storage, transformation, and serving. Data products can take many forms, such as reports, dashboards, machine learning models, or APIs.

Figure B-1 illustrates a general structure for designing a data product using the Data Product Canvas. This framework supports scalable data governance by guiding the creation of consistent, reliable, and well-documented data products. Each data product consists of eight key building blocks:

  1. Domain name

  2. Data product name

  3. Consumer and use case

  4. Data contract (more on this in a moment)

  5. Documentation of data sources

  6. Data product architecture, including transformation pipelines and lineage

  7. Ubiquitous language, to establish a shared understanding of terminology

  8. Classification of the data product (source-aligned, aggregate, or consumer-aligned)

Data Contracts

A critical component of data products from a data governance perspective is the data contract. Data contracts are formal agreements between data producers (e.g., source system owners, data engineers) and data consumers (e.g., analysts, data scientists, business users) that define the expectations, responsibilities, and standards for data exchange. They ensure that data consistently meets specific agreed-upon quality, format, and delivery requirements.

In their book Data Contracts (O’Reilly), Chad Sanderson and Mark Freeman define these contracts as “a mechanism for expanding software-oriented collaboration to data teams, bringing quality to data through human-in-the-loop review, just as the same systems facilitated code quality for product teams.”

The key components of a data contract that will ensure effective data governance for your data product include:

Data schema

The data contract should clearly define the structure, types, and relationships between the data fields. It should link to relevant data policies and standards and specify metadata requirements for each field.

Quality standards

The data contract should establish standards for data accuracy, completeness, and consistency. It should also assign the roles responsible for data quality monitoring and improvement.

Delivery schedule

The data contract should specify when and how frequently data will be delivered, ensuring alignment with established data retention policies. Include governance-approved processes for handling late or missing data.

Service level agreements (SLAs)

The data contract should define performance metrics and response times for data delivery and issue resolution and incorporate governance-approved metrics for data availability and accessibility. It should include escalation procedures for SLA violations and processes for governance review and approval of SLA changes.

Security, privacy, and governance

The data contract should ensure that data handling complies with relevant privacy laws and security protocols. Reference specific data classification levels as defined by governance policies and include requirements for data masking and encryption of sensitive data. Specify access control policies and procedures, data handling guidelines for different privacy classifications, and relevant compliance requirements (e.g., GDPR, CCPA, HIPAA). Clearly define the roles of data owners, stewards, and other governance stakeholders.

Change management

The data contract should outline procedures for managing schema changes, data source modifications, and other alterations. Include governance-approved change control procedures and specify required approvals for schema changes or data transformations. Define communication protocols for notifying stakeholders of changes and processes for updating related documentation and metadata.
