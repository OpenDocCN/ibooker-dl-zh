# 前言

我已经数不清有多少次被问及，“LLM/AI 工程师和 LLMOps 工程师之间的区别是什么？”这是一个经常出现的问题，无论我是在会议中、在会议上，还是在与该领域的某个人喝咖啡时。

我过去总是从解释这些角色的技术区别开始。但随着时间的推移，我意识到真正的问题：人们并没有完全理解在长时间的生产环境中保持大型语言模型（LLM）运行所需的条件。

当我于 2025 年初写下这些内容时，顶级模型、技术和最佳实践每几天都在变化。因此，很少有人理解它们的复杂性。大多数人仍然认为“运营化”，或“Ops”，就是部署，但在 LLM 的背景下，Ops 实际上是指通过简化人员、流程和技术，使这些模型在生产中更加安全、健壮和可靠。

企业及其人力资源部门都在努力弄清楚这一切对他们团队和项目意味着什么，在这本书中，我已经尽我所能回答了这个问题。这本书不是关于定义角色或如何构建和部署 LLM 的教程；虽然它涉及这两个主题，但这已经不够了。一旦基于 LLM 的应用进入生产，就必须有人保持其优化，否则它们可能会变成过度工程化的简单问题的解决方案，或者更糟糕的是，成为在高度需求或提示注入攻击下崩溃的糟糕维护的纸牌屋。

在传统的软件开发（或软件 2.0）中，你不会要求你的首席开发者来构建和维护你的整个产品。软件开发工程师负责构建，可靠性工程师负责维护。构建和维护大型语言模型（LLM）需要类似的职责分离。在软件 3.0 中，LLM/AI 工程师负责构建，而 LLMOps 工程师负责维护！

尽管机器学习运营（MLOps）是 LLMOps 的基础，但工程师从处理结构化数据和判别性模型中获得的能力并不能完全转移到生成模型。

简而言之，我写这本书是为了帮助你欣赏基于 LLM 的全生命周期应用的独特方面，从数据工程到模型部署，再到 API 设计、监控、安全和资源优化。我希望为你提供一个强大的基础，以便你在构建、维护和优化你的 LLM 数据、模型和应用时做出决策。
